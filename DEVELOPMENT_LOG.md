# 图片工具开发过程记录

> 开始时间：2026-07-16（Asia/Shanghai）  
> 记录规则：从本次开发开始，持续记录每次操作、目的、结果和下一步；涉及代码修改时列出主要文件，涉及验证时记录实际结果。

## 2026-07-16

### 01. 确认开发约定

- 操作：确认开始开发移动端优先的图片格式转换、压缩、体积调整、分辨率修改和模板裁剪网站。
- 目的：将已确认的浅色/深色移动端原型转化为可运行产品。
- 结果：采用现有 `PROJECT_ARCHITECTURE.md` 与 `prototypes/mobile-prototype-light-dark-v2.png` 作为产品和视觉基线。
- 约定：后续命令、修改、验证、问题和决策均追加到本文档。

### 02. 读取网站构建规范

- 操作：读取 `sites:sites-building` 技能说明。
- 目的：按当前 Sites 项目的标准结构完成初始化、开发、验证与后续上线。
- 结果：确认本项目属于带图片上传/处理能力的 capability path；首先初始化站点并启动本地预览，然后开发移动端产品界面与本地图片处理能力。

### 03. 初始化站点工程

- 操作：使用 Sites 官方初始化脚本在当前目录创建项目并安装基础依赖。
- 目的：获得标准的可开发、可构建和可部署站点结构。
- 结果：初始化器在写入前检测到当前目录非空并安全退出，未修改现有文件。
- 问题：架构文档、过程记录和原型图使初始化器无法直接创建项目。
- 恢复方案：将三项现有资料临时移动到 `/tmp/pic_tools_preinit`，在原目录重试初始化，完成后原样恢复。

### 04. 临时保护已有资料

- 操作：临时移动 `PROJECT_ARCHITECTURE.md`、`DEVELOPMENT_LOG.md` 和 `prototypes/`。
- 目的：在不丢失既有成果的前提下让初始化器使用空目录。
- 结果：已有资料已安全暂存；初始化脚本随后成功创建站点并安装 509 个基础依赖；资料已全部恢复到项目根目录。
- 说明：安装过程只有两个上游包的 deprecated 警告，没有安装失败。

### 05. 启动本地开发预览

- 操作：启动站点开发服务并保留进程。
- 目的：让后续界面修改可以即时预览，同时尽早确认项目可运行。
- 结果：首次启动被受限环境阻止监听调试端口 `0.0.0.0:9229`，项目代码本身尚未报错。
- 恢复方案：使用已允许的站点开发命令在受限环境外重新启动。
- 最终结果：开发服务已成功运行，本地地址为 `http://localhost:3000/`。
- 预览说明：当前会话没有提供 Sites 规范中提到的应用内打开工具，因此暂未自动打开页面；服务保持运行。

### 06. 检查初始化后的核心文件

- 操作：读取页面、布局、全局样式、依赖配置和 Sites 托管配置。
- 目的：确认现有项目结构与可复用依赖，再一次性实现首版交互式产品。
- 结果：确认项目为 Next.js/Vinext + React + TypeScript + Tailwind；当前页面仍是临时加载骨架，托管配置未启用数据库或对象存储。
- 决策：首版采用纯浏览器 Canvas 处理，图片不上传服务器；实现 JPG/PNG/WebP 输出、目标体积压缩、分辨率和模板裁剪。HEIC/AVIF 等受浏览器解码/编码能力限制的格式先做清晰提示，后续再接隔离服务端 Worker。
- 视觉：以已确认的浅色/深色移动端原型为准，并在桌面宽屏上增加居中工作台，而不是复制桌面管理后台。

### 07. 调整界面依赖

- 操作：移除仅供初始骨架使用的 `react-loading-skeleton`，加入统一线性图标库 `lucide-react`。
- 目的：删除不再需要的启动页依赖，并保证功能图标风格一致、可访问且易维护。
- 结果：`react-loading-skeleton` 已移除。首次安装 `lucide-react` 时受限网络无法解析 npm 地址，获得授权后重试成功。
- 依赖审计：npm 报告现有依赖树有 16 个漏洞（1 low、9 moderate、6 high）；本轮不执行可能引入破坏性升级的 `npm audit fix --force`，在功能完成后单独核查来源和可修复范围。

### 08. 实现首版产品

- 操作：替换临时骨架，编写移动端优先的完整单页工具、浅色/深色主题和浏览器端图片处理逻辑；同步更新站点元数据与全局设计变量。
- 目的：形成从上传到编辑、模板裁剪、处理结果和下载的完整可用闭环。
- 计划修改：`app/page.tsx`、`app/globals.css`、`app/layout.tsx`；移除不再引用的 `_sites-preview` 文件。
- 结果：已完成上传、格式转换、目标体积压缩、分辨率调整、模板裁剪、前后对比、结果下载、响应式导航和浅色/深色主题。
- 本地处理：使用浏览器 Canvas 完成处理；JPEG 输出自动铺白色背景，模板使用居中 cover 裁剪，JPEG/WebP 目标体积通过质量二分搜索逼近。
- 兼容提示：首版明确提示 30 MB 上限以及 HEIC/RAW 等浏览器无法解码格式的后续服务端支持计划。
- 清理：已删除临时骨架组件和样式，替换默认标题、描述与页面语言。
- 自检修正：避免结果 URL 更新时错误释放原图 URL；模板入口会先切换到裁剪工具再执行处理。

### 09. 检查代码变更并执行生产构建

- 操作：查看工作区变更摘要，然后运行完整生产构建。
- 目的：发现类型、编译和 Cloudflare/Vinext 输出兼容问题。
- 结果：生产构建成功，客户端、服务端、RSC、SSR 五个阶段均通过；共转换客户端 1813 个模块。
- 提示：Vinext 对根路由给出静态/动态分类尚不确定的非阻断说明，不影响构建产物生成。
- Git 状态：当前为新初始化项目，文件均尚未提交；未覆盖或丢失用户已有文档与原型。

### 10. 验证页面响应与自动测试

- 操作：请求正在运行的本地首页，并读取后执行项目内置页面测试。
- 目的：确认开发服务已加载新页面，而不只是构建器成功。
- 页面响应：生产渲染实际返回 200，并包含中文站点标题、首页上传入口、四项工具和隐私说明，证明新页面已进入构建产物。
- 测试结果：初始化器自带的两个测试仍专门断言“临时加载骨架必须存在”，因此在骨架按规范删除后均失败；失败内容与产品代码无关。
- 修复方案：将启动器测试改为正式产品验收测试，检查中文元数据、核心入口、隐私文案、无临时标记、主题变量和处理逻辑存在。

### 11. 更新正式产品测试

- 操作：替换 `tests/rendered-html.test.mjs` 中仅适用于启动骨架的断言。

- 结果：两个正式产品测试全部通过（2 passed，0 failed）。
- 覆盖范围：服务端中文页面、产品标题、四项核心工具、隐私说明、临时骨架清理、Canvas 处理逻辑、目标体积、模板、主题持久化、移动断点与减少动效偏好。
