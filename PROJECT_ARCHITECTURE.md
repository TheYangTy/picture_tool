# 图片格式转换与尺寸处理工具：项目架构文档

> 文档状态：架构基线 v0.1  
> 编写日期：2026-07-16  
> 项目目标：以最小可行版本快速上线，通过搜索与自然分享获取用户，同时保留扩展为批量、会员与 API 产品的能力。

## 1. 产品定义

这是一个面向普通用户、设计师、自媒体、电商运营和开发者的在线图片处理工具。用户无需安装软件，即可完成：

1. 图片格式相互转换；
2. 压缩图片体积；
3. 增大图片体积；
4. 修改宽高、百分比缩放和 DPI；
5. 按主流软件、内容平台和证件照模板一键裁剪；
6. 批量处理并打包下载。

产品的核心价值不是单纯“功能多”，而是：

- 打开即用，不要求注册；
- 上传后自动识别格式，默认给出合理结果；
- 处理前后可对比画质、分辨率和文件体积；
- 对隐私、删除时间和格式兼容性做清晰说明；
- 每个高频需求都有独立、可被搜索引擎收录的入口页。

### 1.1 产品边界

首版不做完整在线 Photoshop，不加入图层、复杂滤镜、文字排版和多人协作。核心流程始终保持为：

`选择图片 → 选择目标 → 调整参数 → 预览 → 下载`

“支持市面上大多数格式”应解释为分层支持，而不是承诺任意格式都能互转：

- 高频格式：完整支持读取、编辑和导出；
- 专业/设备格式：优先支持读取并转换为通用格式；
- 文档/矢量格式：受控地转换首页或指定页面；
- 少见、存在专利/授权或安全风险的格式：按插件逐步开放。

## 3. 关键产品决策

### 3.1 混合处理架构

采用“浏览器本地处理 + 服务端兼容处理”的混合模式。

```text
用户选择文件
   │
   ├─ 常见格式、文件较小、浏览器能力满足
   │      └─ 浏览器 Worker / WASM 处理 → 本地下载
   │
   └─ HEIC/TIFF/RAW/PSD/PDF、动画、大文件或浏览器不支持
          └─ 上传临时对象存储 → 隔离 Worker 处理 → 限时下载
```

这样做的原因：

- 本地处理隐私更好、边际成本低、交互快；
- 服务端负责浏览器无法稳定解码的格式；
- 同一套 `TransformSpec` 描述转换参数，避免两套产品行为分叉；
- 首版可先实现服务端稳定路径，再逐步把高频任务下沉到浏览器。

### 3.2 增大图片体积的定义

产品必须避免让用户误以为“文件更大等于画质变好”。提供三种模式：

1. **提高导出质量**：提高 JPEG/WebP/AVIF 质量参数，尽量保留现有细节；
2. **放大分辨率**：使用 Lanczos 等传统算法放大，未来可增加 AI 超分辨率；
3. **达到目标体积**：在不改变可见内容的前提下，通过高质量重编码和格式合法的填充块，使结果达到用户指定的最小 KB/MB。

目标体积算法：

- “不超过 N KB”：在合法质量范围内二分搜索，必要时再缩小分辨率；
- “尽量接近 N KB”：在质量、色度抽样和压缩级别间搜索最接近结果；
- “至少 N KB”：先以最高合理质量导出，不足部分使用对应格式允许的非关键元数据/注释块填充；
- UI 同时展示“像素是否变化”“可见质量是否变化”和“额外填充量”；
- 不对不适合安全填充的格式承诺字节级精确值。

### 3.3 裁剪模板不是硬编码页面

所有模板进入版本化 `Preset Registry`：

```json
{
  "id": "social.example.cover",
  "name": "示例平台封面",
  "category": "social-cover",
  "width": 1200,
  "height": 630,
  "fit": "cover",
  "safeArea": { "top": 40, "right": 40, "bottom": 40, "left": 40 },
  "sourceUrl": "https://example.com/official-spec",
  "verifiedAt": "2026-07-16",
  "status": "active",
  "version": 1
}
```

模板后台可修改、下线和保留历史版本。平台规格可能变化，因此每条模板都需要来源、核验日期和版本号。首批分类建议：社交媒体封面、视频封面、电商主图、演示文稿封面、桌面与手机壁纸、常见证件照、自定义尺寸。

## 5. 图片处理领域模型

### 5.1 统一转换协议 `TransformSpec`

前端、本地引擎、API 和 Worker 必须共享版本化协议：

```ts
type TransformSpecV1 = {
  version: 1;
  input: {
    assetId?: string;
    preserveAnimation: boolean;
    page?: number;
  };
  operations: Array<
    | { type: "autoOrient" }
    | { type: "crop"; x: number; y: number; width: number; height: number }
    | { type: "resize"; width?: number; height?: number; fit: "contain" | "cover" | "fill" }
    | { type: "flatten"; background: string }
    | { type: "stripMetadata"; keepIcc: boolean }
  >;
  output: {
    format: "jpeg" | "png" | "webp" | "avif" | "gif" | "tiff";
    quality?: number;
    targetBytes?: { mode: "max" | "nearest" | "min"; value: number };
    filename: string;
  };
};
```

处理顺序必须固定：识别真实文件类型 → 解码 → EXIF 自动旋转 → 选择帧/页 → 裁剪 → 调整分辨率 → 色彩空间转换 → 编码 → 目标体积优化 → 元数据策略 → 输出校验。

### 5.2 能力协商

每个引擎暴露：

- 可读取格式；
- 可写入格式；
- 是否保留透明通道；
- 是否保留动画；
- 最大文件字节数、像素数、帧数和页数；
- 可用操作与版本；
- 当前运行环境的 delegate/codec 构建信息。

前端根据 `GET /v1/capabilities` 展示真实能力。任务路由器先尝试本地引擎，其次高频服务端引擎，最后扩展引擎。

## 6. 格式支持策略

### 6.1 首版格式矩阵

| 格式 | 输入 | 输出 | 动画 | 引擎 | 备注 |
|---|---:|---:|---:|---|---|
| JPEG/JPG | 完整 | 完整 | 不适用 | Sharp/浏览器 | 支持渐进式与质量控制 |
| PNG/APNG | 完整 | PNG 首发 | APNG 后续验证 | Sharp/扩展 | 透明、调色板和无损优化 |
| WebP | 完整 | 完整 | 尽量保留 | Sharp | 静态与动画要分别测试 |
| AVIF | 完整 | 完整 | 首版静态 | Sharp | 编码较慢，设置更严格超时 |
| GIF | 完整 | 完整 | 保留或抽取首帧 | Sharp | 大动画资源消耗高 |
| TIFF | 完整 | 完整 | 多页受限 | Sharp | 首版默认第一页 |
| SVG | 受控输入 | 暂不输出 | 不适用 | Sharp/隔离渲染 | 必须清理外链、脚本与实体 |
| HEIC/HEIF | 输入 | 条件输出 | 首版静态 | 扩展 Worker | 取决于构建 codec 和授权环境 |
| BMP/ICO | 输入 | P1 输出 | ICO 多尺寸受限 | 扩展 Worker | 常见兼容需求 |
| PDF | P2 输入 | 否 | 多页 | 扩展 Worker | 仅栅格化；严格沙箱 |
| PSD/EPS/RAW | P2 输入 | 否 | 合成预览 | 插件 Worker | 不承诺图层/编辑语义 |

Cloudflare 的公开能力清单也体现了行业常见的“输入格式多于输出格式”策略，并对文件大小、总像素和动画总像素设置限制；本项目应采用同类资源保护思路，而不是允许无限制解码。[Cloudflare Images 限制与格式](https://developers.cloudflare.com/images/get-started/limits/)

### 6.2 格式降级提示

转换前必须提示可能丢失的能力：

- 透明图片转 JPEG：选择背景色，默认白色；
- 动画转静态：选择首帧、指定帧或拼图；
- HDR/广色域转普通格式：提示色域变化；
- 多页 TIFF/PDF：选择页码或输出 ZIP；
- SVG/PSD 转位图：提示矢量、图层和文字可编辑性将丢失；
- 元数据清理：默认删除 GPS，是否保留 ICC 单独控制。

## 7. 核心模块

### 7.1 Web 与编辑器

- 文件拖放、粘贴、移动端相册选择；
- 自动识别格式、尺寸、体积、透明/动画状态；
- 原图/结果滑动对比、100% 像素查看；
- 裁剪框、缩放、旋转、主体位置和安全区叠层；
- 单张参数与批量应用参数；
- 估算输出体积，最终以实际编码为准；
- 本地任务与云端任务使用一致的进度状态。

### 7.2 任务编排

任务状态：

`created → uploading → queued → processing → validating → ready → expired`

异常状态：`rejected | failed | cancelled`

要求：

- API 创建任务时完成参数、配额和能力校验；
- 使用幂等键避免重复扣配额和重复处理；
- Worker 必须可安全重试；
- 心跳和租约防止任务永久卡在 processing；
- 结果必须重新解码校验，不能只凭进程退出码判断成功；
- ZIP 打包作为独立任务，避免阻塞图片转换 Worker。

### 7.3 模板注册中心

- 维度：平台、用途、横竖方向、地区、状态；
- 支持安全区、圆形头像预览和移动/桌面差异；
- 后台修改需审核并保留变更记录；
- 每周自动标记超过核验周期的第三方平台模板；
- 自定义模板可保存到用户账户，匿名用户保存在浏览器。

### 7.4 配额与商业化

匿名用户：无需登录、较小文件、较少批量数、标准优先级。  
免费账户：历史记录、自定义模板、更高批量上限。  
付费账户：大文件、更高并发、长尾格式、长期下载、无广告。  
API 客户：按处理次数、输入像素或计算量计费。

首版不要用付费墙阻挡第一次成功下载。用户先感受到价值，再在大文件、批量和高级格式上升级。

## 8. API 草案

| Method | Path | 用途 |
|---|---|---|
| GET | `/v1/capabilities` | 返回当前真实格式与操作能力 |
| GET | `/v1/presets` | 获取尺寸模板，支持分类和版本 |
| POST | `/v1/uploads` | 创建预签名上传 |
| POST | `/v1/jobs` | 创建转换任务 |
| GET | `/v1/jobs/:id` | 查询状态和进度 |
| POST | `/v1/jobs/:id/cancel` | 取消任务 |
| GET | `/v1/jobs/:id/download` | 获取限时下载地址 |
| POST | `/v1/batches` | 创建批量处理 |
| GET | `/v1/batches/:id` | 查询批量状态 |

`POST /v1/jobs` 不直接接受任意命令行参数，只接受经过 schema 白名单校验的 `TransformSpec`。客户端传入的 MIME、扩展名、宽高和文件体积都不可信，服务端必须重新探测。

## 9. 数据模型

核心表：

- `users`：账户、方案、地区和状态；
- `assets`：对象 key、探测格式、字节、宽高、帧数、过期时间；
- `jobs`：spec 版本、状态、进度、错误码、引擎与耗时；
- `job_outputs`：结果格式、字节、宽高、校验状态和过期时间；
- `batches` / `batch_items`：批量关系与聚合进度；
- `presets` / `preset_versions`：模板及版本；
- `usage_ledger`：可审计的配额增减记录；
- `conversion_metrics_daily`：聚合后的格式成功率和耗时，不保存原始图片内容。

匿名首版可以不为每个用户建账户，但仍需要匿名 session、任务与滥用控制。原始文件名属于用户数据，日志中应散列或脱敏。

## 10. 安全、隐私与资源保护

图片解码属于高风险输入处理，安全要求是架构的一部分：

- 通过文件签名和实际解码探测格式，不信任扩展名/MIME；
- 限制压缩前字节、解码后总像素、宽高、帧数、页数、ICC/EXIF 大小；
- 防范 decompression bomb、畸形文件、超深 SVG、外部实体和远程资源；
- Worker 容器使用非 root、只读文件系统、临时目录、无出站网络；
- 按任务设置 CPU、内存、打开文件数和墙钟超时；
- ImageMagick 使用最小 policy，禁用不需要的 coder/delegate；
- 上传和下载 URL 短时有效，bucket 默认私有；
- 原图与结果默认 1 小时内删除，失败任务立即进入清理队列；
- 日志不记录图片二进制、下载签名或完整用户文件名；
- 对匿名 IP/session、账户和任务成本同时限流；
- 依赖与原生 codec 镜像定期更新、扫描和回归测试。

## 11. 性能与可靠性指标

首版 SLO 建议：

| 指标 | 目标 |
|---|---|
| 页面核心交互可用 | 移动网络下尽量小于 2.5 秒 |
| 常见格式任务成功率 | ≥ 99%（排除损坏/超限输入） |
| 10 MB 静态图处理 | P95 小于 8 秒 |
| 任务状态可见延迟 | 小于 2 秒 |
| 临时文件按时删除 | ≥ 99.99% |
| Worker 卡死发现 | 小于 60 秒 |

关键指标：上传转化率、处理成功率、成功下载率、再次使用率、格式对失败率、P50/P95 耗时、每百万像素成本、模板使用率、自然搜索落地页转化率。

## 12. 快速获客与 SEO 架构

不要只做一个 `/tool` 页面。用同一处理核心生成意图明确的静态/服务端渲染页面：

```text
/convert/jpg-to-png
/convert/heic-to-jpg
/compress/image
/compress/jpeg-to-200kb
/resize/image
/resize/instagram-cover
/crop/youtube-thumbnail
/formats/avif
```

每个页面包含：用途说明、立即使用入口、支持限制、隐私说明、常见问题、相关工具和结构化数据。页面内容必须针对任务真实编写，不能批量生成只有关键词不同的空壳页。

增长闭环：

1. 用户从具体需求页进入；
2. 不注册完成第一次处理；
3. 下载页提供相关任务，如“再压缩到 200 KB”“裁成平台封面”；
4. 批量、大文件和保存模板触发注册；
5. 聚合匿名失败数据，优先补齐最常见格式与模板。

## 13. 上线阶段

### Phase 0：技术验证（3～5 天）

- 建立格式 fixture 集和真实能力检测脚本；
- 验证 Sharp 对 JPEG/PNG/WebP/AVIF/GIF/TIFF/SVG 的读取和输出；
- 验证目标体积二分搜索、透明转 JPEG、EXIF 旋转和动画限制；
- 在目标部署环境确认 HEIC codec，不能只在开发机验证。

退出条件：格式矩阵、耗时、峰值内存和失败类型有实测结果。

### Phase 1：MVP（约 2～3 周）

- 无登录单张/批量上传；
- JPG、PNG、WebP、AVIF 转换；
- JPEG/WebP/AVIF 质量压缩、PNG 优化；
- 自定义宽高、等比缩放、基础裁剪；
- 20～30 个高频模板；
- 临时对象存储、任务进度、下载和自动删除；
- 8～12 个高质量 SEO 落地页；
- 错误分类、基础观测、限流和隐私页面。

### Phase 2：兼容与留存（约 2～4 周）

- HEIC、TIFF、GIF 动画、SVG、BMP、ICO 输入；
- 目标 KB/MB 与体积增大；
- 浏览器本地处理常见格式；
- 账户、历史记录、自定义模板；
- 模板管理后台与规格核验流程。

### Phase 3：商业化

- 大文件与更高并发；
- RAW/PSD/PDF 等专业输入插件；
- API key、Webhook、用量与账单；
- 团队模板和品牌预设；
- 根据真实需求评估 AI 超分辨率，而不是首版即引入 GPU 成本。

## 14. 测试策略

### 14.1 Fixture 维度

- 每种格式的最小、典型、超大和损坏文件；
- 透明、灰度、CMYK、ICC、EXIF 旋转、HDR/16-bit；
- 静态、多帧、多页、超大元数据；
- 文件扩展名与真实类型不一致；
- SVG 外链、脚本、实体和极端 path；
- 已知 decompression bomb 与超时样本。

### 14.2 验证方式

- 单元测试：spec 校验、尺寸计算、模板安全区、目标体积搜索；
- 兼容测试：输入 × 操作 × 输出矩阵；
- 视觉回归：感知哈希/SSIM + 关键样本人工检查；
- 属性测试：输出可解码、尺寸正确、体积约束成立、透明规则成立；
- 压力测试：并发大图、动画、多页和取消任务；
- 安全测试：超限、畸形和恶意样本不能影响其他任务。

## 15. 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| “多数格式”承诺过宽 | 用户预期失控 | 公布动态能力矩阵，区分输入/输出/动画/多页 |
| AVIF/动画编码昂贵 | 超时和成本上升 | 独立队列、像素/帧限制、较低并发 |
| 原生库安全漏洞 | 文件处理服务被攻击 | 隔离容器、无外网、最小 coder、持续升级 |
| 目标 KB 无法精确命中 | 用户不满 | 给出误差范围，区分最大/接近/最小模式 |
| 平台模板过期 | 裁剪结果不合规 | 来源、核验时间、版本化和下线机制 |
| SEO 页面低质 | 无法获客甚至受惩罚 | 少而精，页面与真实工具状态和限制一致 |
| 云端处理成本失控 | 免费用户不可持续 | 本地下沉、成本限流、像素计量、分级队列 |

## 16. 首版明确不做

- 不做完整图层编辑器；
- 不承诺所有格式互相输出；
- 不做 AI 生成和修图；
- 不永久保存匿名用户图片；
- 不在主应用进程直接执行任意 ImageMagick 命令；
- 不为追求“精确体积”生成可能被目标软件拒绝的非法图片；
- 不在没有官方来源和核验日期时宣称模板尺寸永远有效。

## 17. 立即执行清单

1. 建立 monorepo 和 `TransformSpec` v1；
2. 建立 50～100 个格式测试样本与能力探测报告；
3. 先实现 Sharp Adapter 和结果重新解码校验；
4. 完成上传、任务、结果下载和定时删除闭环；
5. 开发统一编辑器：转换、质量、分辨率、裁剪；
6. 建立模板注册表，录入首批高频模板并保存来源；
7. 发布首批 SEO 任务页；
8. 用真实失败率决定第二批格式，而不是凭格式列表堆功能。

## 18. 架构验收标准

满足以下条件后，项目可以进入正式开发：

- 产品对“压缩”“增大”“分辨率”“DPI”“裁剪”的定义无歧义；
- 格式矩阵明确区分输入、输出、动画、多页和降级行为；
- 所有处理入口共用 `TransformSpec`，没有页面专属转换逻辑；
- 扩展解码器与 Web/API 进程隔离；
- 每个任务有资源上限、超时、取消、重试和清理策略；
- 模板可版本化并能追踪来源与核验日期；
- MVP 范围能在 2～3 周形成可获取自然流量的完整闭环。
